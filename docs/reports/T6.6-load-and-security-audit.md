# T6.6 — Load test & security audit (G6 input)

**Date:** 2026-07-23 · **Harness:** `fluxum-bench` (release) · **Machine:** AMD
Ryzen 9 7950X3D (32 logical cores), 128 GiB, Windows 10 · **Server:**
`fluxum-server` release, development profile.

This report is the G6 deliverable for the throughput benchmark (NFR-01),
the fan-out latency measurement (NFR-04), and the adversarial security pass
(TST-070..TST-072). The Grafana dashboard criterion is met by
`ops/grafana/fluxum-overview.json` (see §4).

## 1. Sustained load test (NFR-01 / TST-060)

`fluxum-bench load` drives the `add_task` small write from N pipelined
connections (SDK-032) for a measured window, and reports throughput **via
the `fluxum_reducer_calls_total` counter delta over the window** — the
method TST-060 mandates — with the errored-call count.

| connections × window | window | calls/s (counter delta) | errored | shard-busy |
|---|---|---|---|---|
| 8 × 8   | 15 s | 45,818 | 0 | 0 |
| 16 × 32 | 60 s | **54,228** | 0 | 0 |

**Verdict: NFR-01's ≥ 100,000 calls/s is NOT met on this box — the ceiling
is measured and attributed, not hidden.** The wall is the shard's
single-writer commit loop at **~15.6 µs/commit sustained** (~64k/s
aggregate), established in `phase0_parity-write-pipelining` (P0-B): the
number is invariant to connection count (8→24) and pipeline depth (8→128),
the reducer queue never fills (zero shard-busy at 512 in flight vs the
TXN-011 default capacity of 1000), and the engine-only microbench
(`txn_commit`, no sockets) runs the same cycle at 22.4 µs serial. Every call
committed — **zero errored calls** — so the shortfall is pure throughput
ceiling, not correctness under load.

The named next lever (recorded in SPEC-013 TST-060) is **batch-draining the
writer loop** — amortizing the per-job channel-recv + timestamp + log-accept
overhead across queued jobs — which is an engine change beyond this task's
scope. NFR-01 remains an open, honestly-quantified gap; it does not block
the G6 metrics/dashboard/audit criteria this task also delivers.

## 2. Fan-out latency (NFR-04 / TST-061)

`fluxum-bench fanout` subscribes M clients to one hot channel, commits chat
messages at a fixed rate, and measures every delivery's commit→receipt
latency from a shared in-process instant epoch.

| subscribers | messages | deliveries | p50 | p99 | max | NFR-04 (p99 < 5 ms) |
|---|---|---|---|---|---|---|
| 200   | 60  | 12,000  | 1.05 ms | 1.89 ms | 2.45 ms | **MET** |
| 1,000 | 100 | 100,000 | 4.08 ms | 8.83 ms | 12.70 ms | **NOT met** |

At the TST-061 target of 1,000 subscribers, p99 is 8.83 ms — over the 5 ms
bound. The cause is the **same measured environmental floor** as the P0-A
fan-out work and the parity e2e gap: Windows-loopback `WSASend` (~40 µs per
socket write) across 1,000 sockets, plus 1,000 subscriber reader threads
contending for the same 32 cores as the server (driver-side scheduling,
F-005/F-011). It is delivery-transport and harness cost, not fan-out
evaluation — the subscription engine still evaluates and encodes each delta
once and shares the bytes (SUB-024), which the 200-subscriber run confirms
comfortably (p99 1.89 ms). A Linux-host or split-box rerun is the named
follow-up to demonstrate NFR-04 environment-clean; on this box the p99
crosses 5 ms between ~400 and ~1,000 subscribers.

## 3. Security audit (TST-070..TST-072) — no P0 findings

The adversarial surfaces are defended in code and pinned by tests; the full
threat model and per-category analysis live in
`docs/analysis/owasp-security/`. This audit confirms **no P0 findings** and
maps each required surface to its enforcing test:

| Surface | Defense | Pinned by |
|---|---|---|
| **SQL injection** (subscription compiler, TST-072) | Only the closed SUB-010/011 grammar compiles; every hostile string is a wire-ready 400, never a panic, never a `CompiledPlan`. Values are parsed positionally, never string-concatenated into a query. | `crates/fluxum-core/tests/sql_injection_corpus.rs` — a broad corpus of classic SQLi payloads, comment/encoding tricks, UNION/subquery/stacked statements, and resource-exhaustion shapes, all rejected. |
| **RLS bypass** (TST-071) | `#[visibility(owner_only(..))]` is enforced server-side on the subscription path (DM-060) — two clients on the same `SELECT` get different rows — and reducers re-check ownership explicitly rather than trusting row visibility (a reducer runs against the whole table). | `admin_authz.rs`, the demo module's `complete_task`/`delete_note` ownership checks, subscription visibility tests. |
| **Auth bypass** (TST-070) | An unauthenticated session accepts only `Authenticate` (AUTH-020); every other message is refused before it reaches state. The admin surface rides the SEC-054 guard (loopback-or-trusted-IP-plus-operator-credential). | `session_router.rs`, `admin_authz.rs`, `connection_abuse.rs`, `http_loopback.rs`, `tcp_loopback.rs`. |
| **Rate-limit bypass** | Per-(identity, reducer) admission (RED-050, e.g. chat 20/s) checked pre-transaction; per-tenant quotas (OPS-060); the SEC-047 per-source query-admission bucket that token rotation cannot refill; the SEC-031 handshake/auth-failure throttle. | `rate_limit.rs`, `query_reducer_bounds.rs`, `tenant_quotas.rs`, `connection_abuse.rs`. |

Residual hardening items (all **P1/P2**, tracked, none blocking G6): a global
failed-auth ceiling + optional per-identity lockout (F-018), an asymmetric
verify-only JWT provider variant (F-019), and enforcing (or rejecting at
schema-load) the `shard_local`/`custom`/`member_of` visibility rules so a
declared rule never silently means "no filter" (F-003). These are the
`docs/analysis/owasp-security/10-execution-plan.md` follow-ups, candidates
for a future `auth-and-rls-hardening` task.

## 4. Metrics dashboard (PRD §12.1)

`ops/grafana/fluxum-overview.json` covers **every P0 `fluxum_*` metric
family** the server exports — liveness/lifecycle, reducer throughput +
duration percentiles (NFR-01/03), writer queue depth, fan-out rate + the
OBS-023 per-stage latency, subscriptions + drops, connections, and the
security surface (auth outcomes, rejections, rate-limits, connguard). A
`shard` template variable filters every panel. Coverage is enforced by
`crates/fluxum-server/tests/dashboard_metrics.rs`, which scrapes a live
server's declared metric families and fails if any lacks a panel — a new
metric cannot ship unmonitored. Provisioning is in `ops/grafana/README.md`.
