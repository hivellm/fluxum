Streamable HTTP (commit `6dcb8cf`): `http://host:port` URLs speak RPC-004..007 — `POST /rpc` carries requests with replies in the response body, the background thread reads the `GET /rpc` chunked push stream, `Fluxum-Session` binds the two. Hand-rolled HTTP/1.1 over `TcpStream` so the SDK keeps zero extra dependencies. Reconnect is transport-aware: an HTTP stream drop is first a BLIP (CS-021 — reattach + `Resume` from applied offsets; the ResumeTracker drives the path it was built for), falling back to full re-establishment on a dead session (404); the chunked decoder drains buffered frames before honouring the terminal chunk so a final 408 error frame is never lost. **The corpus runs 22/22 over BOTH transports** (SPEC-006 acceptance 6 through the SDK), plus two HTTP e2e tests (full session, blip recovery without update loss).
- [x] 1.3 Local client cache + reconnect/resubscribe/reconcile behavior consistent with SDK-04x rules — **complete** (commit `d23516b`). Cache: `RowCache` byte-identity refcount, per-query attribution, delete+insert coalescing (SDK-042/044/045) **plus the reconnect trio ported from the TS cache**: `reconcile` (net-difference rebuild — unchanged rows fire nothing, refcounts rebuilt from snapshot duplicates, a table absent from the snapshots is unsubscribed and cleared), `reset_queries`, `track_query` (re-attribution under the new server ids); 9 unit tests. Client: the reader thread becomes the reconnect loop on disconnect — connect, authenticate, resubscribe, reconcile, in that order (reconciling first would read the coverage gap as deletes) — exponential backoff with jitter (`ReconnectPolicy`, public; defaults match the TS SDK; `connect_with` overrides, `Drop` wakes a sleeping loop via condvar). Subscribe handles are stable across reconnects: the client re-points them at the fresh server-assigned query ids internally (`SubEntry {sql, app_id, server_id}`), so `unsubscribe`/`applied_offset` keep working after a resync.
- [x] 1.3b Wire `fluxum_sdk::ResumeTracker` into the real connection (SPEC-021 CS-020/CS-021/CS-022) — **fed from the live socket** (commit `f794b0a`): every `InitialData`/`TxUpdate` the Connection applies advances the tracker's per-subscription `tx_offset` (CS-020), exposed via `Connection::applied_offset`; `cache_reset` is honoured on the application path (CS-022 — the query's rows are cleared before a reset snapshot replaces them). **Scope correction:** the `resume_message()`-instead-of-resubscribe path is the *session-preserving* reconnect (the HTTP GET-stream blip). A full TCP reconnect is a NEW session whose query_ids the server does not recognise, so that client must resubscribe+reconcile (as the TS SDK does), not Resume. The auto-reconnect loop landed with 1.3 (commit `d23516b`) and does exactly that: resubscribe+reconcile over TCP; the tracker is rebuilt from the new session's snapshot (its offsets restart with the session).
