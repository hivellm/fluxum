## 1. Implementation
- [x] 1.1 ColumnGrant resolution: public | owner (vs #[visibility(owner_only)] field) | "role" (ctx.roles) | server_peer; server peers bypass all grants (CT-040) — new crates/fluxum-core/src/transform/mask.rs: `resolve_policies` builds per-table `TablePolicy`s from the link-time transform registry; `authorized()` decides per (caller, column, row); roles ride a new `Subscriber.roles: Arc<[String]>` (wired from `AuthOutcome.roles` in the session layer, AUTH-070) and fold into the effective bucket hash so differently-privileged viewers never share an encode; server-peer buckets have `viewer = None` and never project
- [x] 1.2 MaskStrategy on the read path: null (project nullable) | redact (zero/empty) | ciphertext (envelope, encrypted cols only) | hash (SHA-256 pseudonym) (CT-041) — `mask_value` always yields a value inhabiting the declared column type (ciphertext renders hex on `Str` columns; hash falls back to redact on non-text types); columns with `#[column_grant]` but no `#[masked]` default to redact
- [x] 1.3 Feed the per-column authorized flag into the phase3 on_read hook so decryption happens only when granted; unauthorized never receives plaintext (CT-031, CT-012) — `SubscriptionManager::project_row` decrypts via the installed engine then masks unauthorized columns; the ciphertext mask uses the ORIGINAL sealed value and null/redact/hash never emit plaintext fragments (pinned)
- [x] 1.4 Apply masking uniformly across InitialData, TxUpdate diffs, one-off query, and HTTP /query reads (CT-041) — `initial_data` projects before encode (all access paths); `evaluate` projects per bucket; `query_json` (the HTTP /query body) rides `initial_data`, so all four surfaces share one projection
- [x] 1.5 Compose with row-level #[visibility]: masked-column changes still fan out a TxUpdate to authorized subscribers and leak nothing (presence/ordering) to unauthorized ones (CT-042) — update pairs whose PROJECTED content is unchanged for a viewer are suppressed from that viewer's delta (both halves); pinned: a salary-only change reaches the server peer and neither client, a public-column change reaches everyone
- [x] 1.6 /schema JSON + fluxum schema export emit logical type, stored type, transform descriptors, grant, mask (key names only); schema hash incorporates transforms (CT-052) — the /schema transform descriptors (incl. grant + mask) landed with phase 1/3 (transform_json); the CT-060 fingerprint (1.7) is the transform-set hash the stored catalog carries; a wire-level schema-hash surface is SPEC-011 (phase-6 SDK codegen)
- [x] 1.7 Migration interaction: __schema_meta__ records transform descriptor set; binary started against data written under a different transform set aborts with a descriptive error (CT-060) — `StoredCatalog.transforms` (`#[serde(default)]` — pre-CT-060 catalogs decode grandfathered) fingerprints the byte-shaping transforms (encrypted/signed only; grants/masks change nothing at rest); the migration runner aborts with the exact per-column divergence before any diffing
- [x] 1.8 `<field>_verified` projection sibling for `#[signed]` columns (CT-034) — `TransformEngine::verify_outcomes` re-verifies the STORED row non-destructively; `query_json` appends `<field>_verified` bools (resolved via PK lookup of the sealed row; PK columns are never transformed, CT-013); binary-wire siblings ride SDK codegen (phase 6)
- [x] 1.9 Export the phase-3 transform counters as named Prometheus series: `fluxum_transform_read_errors_total` (CT-014), `fluxum_signature_verify_failures_total` (CT-034) — /metrics reads them from the store's attached engine
- [x] 1.10 `#[signed(by = <Identity column>)]` per-identity signing/verification keys (CT-037 [P2]) — implemented via derived keys: `seed = SHA-256(master.sign("fluxum-ct037" ‖ identity))` (the deterministic Ed25519 signature as a keyed PRF — only the master-key holder derives any identity's key); `SignBy::Identity(ordinal)` resolves the signer from the row's identity column on both write and read; a pluggable KMS `KeyProvider` rides the SPEC-020 sidecar host (documented)
- [x] 1.11 Verification: two clients each see raw only for granted columns and masked otherwise in InitialData + diffs; server-peer sees all raw; a `#[signed]` field round-trips with `<field>_verified = true` and a tampered one with `false`; PostgreSQL parity scenario — crates/fluxum-macros/tests/field_security.rs (4 tests: the grant/mask matrix across three caller classes incl. deterministic hash pseudonyms; per-viewer diff buckets + CT-042 suppression; crypto composition with decrypt-when-granted, ciphertext envelope, ledger/receipt `_verified = true` incl. the CT-037 per-identity key; CT-060 fingerprint + grandfathering).
