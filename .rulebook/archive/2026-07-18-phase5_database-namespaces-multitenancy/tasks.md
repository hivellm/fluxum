## 1. Implementation
- [x] 1.1 Introduce a namespace registry: named databases each owning an independent MemStore + TxPipeline + schema, created/looked up by name; a default namespace preserves single-DB behavior (OPS-050) — crates/fluxum-server/src/namespace.rs (Namespace owns engine + subscriptions + commit broadcast) plus ShardContext::register_namespace/resolve_namespace/namespaces. The ShardContext's own engine/subscriptions ARE the default database, so registering nothing behaves exactly as before and ~90 existing call sites stayed untouched
- [x] 1.2 Namespace selection on connect: a session names its database at auth/connect and is bound to it for the connection lifetime (OPS-050) — tail-additive `namespace: Option<String>` on the Authenticate message; an unknown name fails the handshake (AUTH_FAILED); a re-Authenticate naming another database is refused rather than switching data under live subscriptions
- [x] 1.3 Route reducer calls, queries, and subscriptions through the connection's namespace so each sees only its own tables (OPS-050) — Session holds the binding and engine()/subscriptions()/store()/publish_commit() route through it; both transports follow it for lifecycle hooks, subscription cleanup and commit publishing (HTTP persists the binding with its per-request session state)
- [x] 1.4 Enforce strict isolation: reject any transaction or subscription that references another namespace with a typed error — no cross-namespace access (OPS-050) — structural rather than a check: no code path takes a namespace name at query time, so cross-namespace access is unrepresentable. Each namespace also runs its own fan-out over its own commit broadcast, so a commit is only ever evaluated against that tenant's subscriptions (pinned by a cross-tenant fan-out test)
- [x] 1.5 Per-namespace identity scope: identities/connection ids are scoped within a namespace, not global (OPS-050) — a connection's identity is only meaningful inside its bound database: its reads, writes, RLS/visibility evaluation and subscription registration all happen in that namespace's store and subscription manager, so the same identity in two namespaces addresses disjoint data (asserted by the two-tenant test)
- [x] 1.6 Attribute metrics per namespace via a namespace label on fluxum_* series, and make checkpoints/backups per-namespace (OPS-051) — Metrics::prometheus_in_namespace adds the label by relabelling the standard exposition (so the two renderings cannot drift as series are added); /metrics appends each namespace's series while the default keeps its original unlabelled set for backward compatibility. Checkpoints/backups are per-namespace by construction — each database is opened over its own store directory and commit log
- [x] 1.7 Verification: with namespaces acme and globex, a client authenticated into acme sees only acme tables and is refused when subscribing to or mutating globex; metrics carry the namespace label — tests/namespaces.rs covers exactly this, plus fan-out isolation, unknown namespace, mid-life switch refusal and duplicate/reserved-name registration
