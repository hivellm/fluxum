## 1. Implementation
- [x] 1.1 Add a drain state to the server: entering drain flips accept loops to refuse new connections and new subscriptions while keeping existing ones serviced (OPS-030; crates/fluxum-server/src/lib.rs, tcp.rs, http.rs) — `ShardContext::{begin_drain, is_draining}` (idempotent, deliberately one-way: a drain precedes exit, so there is no "un-drain" to race a shutdown). Entering drain also sets `ShardState::ShuttingDown`, so `/health` goes 503 and a load balancer pulls the shard from rotation. TCP refuses new connections at accept (closing *is* the retryable signal on a raw socket). **HTTP deliberately does not refuse at accept** — see the notes below.
- [x] 1.2 Refuse new reducer calls during drain with a retryable error code (not a hard drop) so clients can retry against the restarted process (OPS-030/OPS-031; crates/fluxum-server/src/session.rs) — the session router refuses `ReducerCall`/`Subscribe`/`SubscribeSingle` with `CLUSTER_SHARD_UNAVAILABLE` (8000: `retryable: true`, HTTP 503). `Unsubscribe`, `OneOffQuery` and `Resume` are deliberately still served: dropping a subscription costs nothing, a read touches no writer, and `Resume` continues an *existing* subscription — refusing them would only hurt the clients a drain is trying to shed gently. The admin `POST /reducer/:name` carries its own refusal, since it reaches the engine directly rather than through the session router.
- [x] 1.3 Quiesce: wait for in-flight transactions to commit before proceeding, tracking outstanding tx count (OPS-030; crates/fluxum-server/src/lib.rs) — `drain()` submits a barrier job to the shard's FIFO single-writer queue: it runs only after every already-admitted call, so **the queue's own ordering is the proof**. No separate outstanding-tx counter is kept, deliberately — a counter is a second source of truth that can drift from the queue, whereas the barrier cannot be wrong.
- [x] 1.4 Trigger a final checkpoint at end of drain so restart replays little or nothing (OPS-030; crates/fluxum-core/src/checkpoint/worker.rs) — `drain(ctx, Some(&SnapshotWorker), opts)` waits for the commit log to fsync the tail, then `observe_commit(durable_tail)` + `checkpoint_now()`. Two details the tests forced out: (a) the worker only snapshots commits it has been *told* about, so the drain names the durable tail explicitly rather than trusting the assembly's feed; (b) the stamp comes from the **commit log**, not `health().last_tx_id` — health tracks what was *published* to the fan-out, the log tracks what is *on disk*. A stamp past the durable tail would make replay skip real commits; one that trusted an unpublishing assembly would under-cover. The durability wait also matters on its own: the log appends asynchronously, so a drain that skipped it could exit having acked writes that never landed.
- [x] 1.5 Bounded deadline: exit cleanly within a configured drain timeout; stragglers past the deadline are force-closed and logged (OPS-030; crates/fluxum-server/src/lib.rs over the existing Arc<Notify> shutdown handle) — `DrainOptions::deadline` (default 30 s) bounds both the quiesce barrier and the durability wait; exceeding it logs a WARN and reports `quiesced: false` rather than erroring, because a straggler must not block a deploy. `drain()` deliberately does **not** stop the transports: the caller drains, inspects the `DrainReport`, then calls the existing `shutdown()` — the force path stays available and unchanged.
- [x] 1.6 Trigger drain from SIGTERM (main) and from a `fluxum drain` command / admin endpoint (OPS-030; crates/fluxum-server/src/main.rs, crates/fluxum-cli/src) — **partially: the triggers ship, the binaries cannot use them yet.** `POST /drain` (the admin-endpoint half) is fully implemented and tested: it enters drain and acks immediately, so a pre-stop hook gets a prompt response and then polls `/health`. `terminate_requested()` provides the signal half — SIGTERM on Unix (what an orchestrator sends before SIGKILL), Ctrl-C elsewhere — as a separate function so the signal source is swappable and testable. **Neither `fluxum-server/src/main.rs` nor `fluxum-cli` can be wired**: both are still T0.1 skeletons that `println!` and exit, with no server assembly to drain and no client to send `fluxum drain` from. Carried as a follow-up on whichever task builds the server assembly / CLI.
- [x] 1.7 Verification: a client with an in-flight reducer call during drain sees that call commit, a call started mid-drain gets the retryable signal, the process checkpoints and exits within the deadline, and restart replay is near-empty — `graceful_drain.rs`: `an_in_flight_call_commits_and_the_drain_checkpoints_it` (the in-flight call commits and the checkpoint reaches the durable tail, so restart replays nothing), `draining_refuses_new_work_with_a_retryable_signal` (asserts the `retryable` flag and 503 status from the SPEC-028 catalog itself rather than hard-coding them), `a_straggler_past_the_deadline_does_not_hang_the_drain`, `draining_an_idle_shard_is_immediate_and_idempotent`; plus `admin_api.rs::drain_endpoint_puts_the_shard_into_drain` end-to-end over HTTP.

**Design notes / limits (deliberate)**
- **HTTP refuses `/rpc`, not connections.** Refusing at the accept loop (as TCP does) would make `/health`, `/metrics` and `/drain` unreachable the moment a drain starts — blinding exactly the load balancer and operator the drain depends on. A test caught this. HTTP therefore accepts, then answers `/rpc` with a retryable 503 while the admin routes keep serving.
- **The quiesce barrier is itself a commit**, so each drain appends one empty transaction and advances the tx id. Drain is idempotent in effect, not a no-op on the log.
- **`drain()` does not exit the process or stop transports** — that belongs to whoever owns the process, and keeping them separate is what lets a caller drain, inspect the report, then force.
- Restart-replay emptiness is verified via the checkpoint stamp reaching the durable tail, not by booting a second process (there is no assembly to boot).
- The drain state is per-`ShardContext`; a multi-shard `ShardCoord` deployment drains each host (its existing `ShardCoord::drain` barrier remains the multi-shard quiesce).
