## 1. Implementation

- [x] 1.1 Define a pluggable SDK persistence backend trait (put/get/delete keyed by `(server, identity, query)`) with persistence opt-in and off by default (CS-040; sdks/rust) — `sdks/rust/src/persist.rs`: `PersistenceBackend` (put/get/delete/list, opaque string keys → bytes). On-disk namespace is `(server, client_id)` with the LAST session's identity stored inside the state (identity is only derived post-auth); a fresh session under a different identity discards the hydrated queue and clears the slice. Off by default: `Connection::connect/connect_with` are untouched.
- [x] 1.2 Implement a native persistence backend (file/SQLite) for the Rust SDK (CS-040; sdks/rust) — `FileBackend`: file-per-key under one directory, hex-encoded names, write-then-rename so a crash mid-write never tears a value; zero new dependencies (file over SQLite deliberately — the crate's dependency surface stays the vendored wire layer alone). Plus `MemoryBackend` as the test double / TS parity twin.
- [x] 1.3 Persist subscribed table state to the local store as authoritative updates apply (CS-040; sdks/rust) — `persist_state` write-through at every mutation point (TxUpdate route, InitialData/resume application, subscribe/unsubscribe, queue changes, post-reconnect adoption); one blob per query (SQL, applied `tx_offset`, held rows via the new `RowCache::query_snapshot`). Locks taken one at a time, never nested.
- [x] 1.4 Persist the pending-mutation queue (with each call's stable idempotency_key) to the local store (CS-040; sdks/rust) — the meta blob carries `OfflineQueue::snapshot()` (keys + `next_seq`, so restored keys never collide) plus the session identity.
- [x] 1.5 On startup, hydrate the cache and queue from the local store before connecting so the UI renders instantly (CS-041; sdks/rust) — `Connection::connect_persistent`: queue restored and hydrated rows seeded into the cache (and the SQLs into the reconnect replay set) BEFORE any network I/O; the TS client hydrates in `connect` before the transport opens.
- [x] 1.6 Reconcile after hydrate: resume via `Resume { from_offset }` (CS-02x) or fall back to a fresh `InitialData` when the offset is gone (CS-041; sdks/rust) — a hydrated client establishes exactly like a reconnect (resubscribe → `adopt_session` net-difference reconcile), reusing the machinery from the optimistic-mutations task. Honest deviation, recorded in SPEC-021's status row: RPC-026 scopes `Resume` to a *surviving session's* query ids, so a fresh post-restart session cannot Resume by wire contract — the reconcile is always the resubscribe + net-diff pass (the sanctioned `InitialData` fallback; `cache_reset` handling already in place). Offsets are persisted for the surviving-session/blip path.
- [x] 1.7 Replay queued mutations through the idempotency key mechanism (CS-03x) so each applies exactly once (CS-041; sdks/rust) — `replay_offline` runs during establishment with the restored queue, in submission order, under enqueue-time keys (server dedup); guarded by the hydrated-identity check so another user's queue is discarded, never replayed as them.
- [x] 1.8 Scaffold the TypeScript SDK persistence backend over IndexedDB for the browser, sharing the hydrate/reconcile/replay flow (CS-040, CS-041; sdks/typescript) — `src/persist.ts`: `PersistenceBackend` interface, `IndexedDbBackend` (lazy-open, one kv object store), `MemoryBackend`, `ClientStore` (msgpack blobs, corrupt-tolerant); `FluxumClient` gained `persistence: { backend, clientId }` with the full flow — hydrate before I/O, identity guard, resubscribe + reconcile (reconnect handlers factored into shared methods), queue replay, write-through.
