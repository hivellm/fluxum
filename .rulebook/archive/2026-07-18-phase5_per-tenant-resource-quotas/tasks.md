## 1. Implementation
- [x] 1.1 Define a per-namespace (or per-identity) quota config: memory-budget share, reducer rate, max concurrent subscriptions, storage bytes; optional per namespace (OPS-060) — TenantQuotas in crates/fluxum-server/src/quota.rs, attached to a Namespace via Namespace::with_quotas; every ceiling optional, an all-None set is exactly the unquotaed namespace
- [x] 1.2 Per-namespace reducer-rate quota layered above the existing per-(Identity, reducer) buckets; exceeding yields a retryable 429-style typed error (OPS-060) — per-namespace token bucket checked at admission in the session's reducer path, above RED-050; breach is REDUCER_RATE_LIMITED (retryable 429). A refused call costs no transaction
- [x] 1.3 Per-namespace memory-budget share: a tenant over its share hits a typed exhaustion error without forcing eviction from other tenants' frames (OPS-060) — the tenant's estimated in-memory footprint (rows × coarse column width, the same figure fluxum_memstore_bytes reports) is checked before admitting a write; refusing the WRITE is what protects neighbours, and no eviction is forced on anyone else because each namespace owns its own store. Implemented at the namespace level rather than inside the shared pager pool, which has no per-tenant frame ownership to divide
- [x] 1.4 Per-namespace subscription-count cap: a new subscription beyond the cap is refused with a typed error (OPS-060) — checked against the tenant's live plan count read from its own SubscriptionManager (so the ceiling cannot drift as connections come and go) and re-checked per query, so a batch cannot vault over it in one request
- [x] 1.5 Per-namespace storage-bytes accounting and cap, with a typed error on exceed (OPS-060) — new CommitLog::disk_bytes() sums the tenant's segment files; sampled through a ~1s cache so a ceiling that moves slowly does not cost a stat per call, and a failed sample reuses the last figure rather than reporting the tenant as empty (which would silently lift its ceiling)
- [x] 1.6 Export usage-vs-quota per tenant as fluxum_tenant_* metrics (rate, memory share, subscriptions, storage bytes) (OPS-061) — fluxum_tenant_{memory_bytes,storage_bytes,subscriptions_active}, fluxum_tenant_quota_bytes{quota} (0 = unlimited) and fluxum_tenant_quota_exceeded_total{quota}, all labelled by namespace and emitted for every quota label even at zero
- [x] 1.7 Verification: tenant A saturating its reducer-rate quota receives 429s while tenant B's latency and admission are unaffected; a memory/subscription/storage over-quota yields the typed error only to the offending tenant; fluxum_tenant_* reflects usage — tests/tenant_quotas.rs covers exactly this (acme throttled while globex writes 20× unaffected and is charged nothing), plus rate refill/recovery, the subscription and memory ceilings refusing only the offender, and an unquotaed namespace behaving as before
