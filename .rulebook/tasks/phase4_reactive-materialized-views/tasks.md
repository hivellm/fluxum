## 1. Implementation
- [x] 1.1 Parse the `materialized` variant of `#[fluxum::view]` (aggregate fn, optional GROUP BY, optional ORDER BY/LIMIT) and register the view definition (RV-010; crates/fluxum-macros/src/reducer.rs) — `#[fluxum::view(materialized, table = T, aggregate = count|sum(col)|avg(col)|min(col)|max(col) [, group_by = col])]` or `(materialized, table = T, order_by = col [, desc|asc], limit = N)` on a unit struct; submits a link-time `MaterializedViewDef` named after the struct; exactly-one-of aggregate/top-N enforced at expansion, unknown args rejected
- [x] 1.2 Compile the view definition into an aggregate/top-N plan over one base table, reusing the SQL compiler surface (RV-010, RV-012) — landed as schema-resolved plans in the new engine (`resolve()` in subscription/matview.rs): column names → ordinals, SUM/AVG numeric-column validation, top-N limit ≥ 1 — the view definition is declarative (no SQL text), so resolution against TableSchema replaced a detour through the SQL parser; descriptive `FluxumError::Schema` on any violation
- [x] 1.3 Add a materialized-view state store keyed by group, holding running aggregate accumulators (RV-010; crates/fluxum-core/src/subscription/matview.rs, new) — `GroupAcc` per memcomparable group key: count, integer/float sums, and a value multiset (memcomparable-keyed BTreeMap) that makes MIN/MAX deletable without re-scans; top-N keeps the FULL ordered multiset `(sort key [inverted for DESC], pk)` so boundary enter/leave is O(log n)
- [x] 1.4 Maintain aggregate state incrementally from commit delta rows, touching only affected groups and never full re-scanning (RV-010; subscription/mod.rs → matview engine) — `on_commit` applies `apply_row(sign ±1)` for exactly the diff's deletes/inserts; `SubscriptionManager::on_commit` feeds the engine on EVERY commit (state correctness independent of subscribers)
- [x] 1.5 Fan out changed view rows to subscribers as `TxUpdate` with cost O(affected groups) (RV-011; crates/fluxum-core/src/subscription/mod.rs) — changed view rows render as `TableUpdate` (view-name table identity): a changed group is delete+insert, a vanished group delete-only; delivered as `QueryDelta`s through the existing fan-out with per-view subscriber lists (pinned: a commit touching one of two groups carries exactly one group row)
- [x] 1.6 Maintain a sorted top-N window and emit enter/leave/reorder deltas as underlying rows change (RV-012; subscription/matview.rs) — view rows `[rank, value, pk-bytes]` keyed by rank: a rank-5→2 jump emits exactly ranks 2–3 (rank 1 untouched); below-window changes emit nothing; leader deletion shifts the window — all bounded by the window, never the table (pinned by the leaderboard test)
- [x] 1.7 Make view state crash-consistent: rebuild from the base table on recovery or validate persisted state against a bit-identical recompute (RV-013; subscription/matview.rs) — state is memory-only: `SubscriptionManager::init_views(&snapshot)` resolves + rebuilds at startup/recovery; `validate_views` asserts incremental state == bit-identical fresh rebuild (exercised after every test scenario)
- [x] 1.8 Expose subscription to a materialized view through the existing subscription registry path (RV-011; crates/fluxum-core/src/subscription/mod.rs) — `subscribe_view(connection, name)` returns the current view rows as `InitialData` (unknown view = wire-ready REDUCER_UNKNOWN_VIEW); `unsubscribe_view` and `disconnect` clean up; view deltas ride `on_commit`'s existing `Vec<QueryDelta>` with a synthetic per-view QueryHash
