## 1. Implementation

- [x] 1.1 Add an optional additive `idempotency_key` field to `ReducerCall` and keep the positional MessagePack encoding stable (CS-030; crates/fluxum-protocol/src/messages.rs) — `#[serde(default)] pub idempotency_key: Option<String>` appended to the **additive tail**. The envelope is positional (`rmp_serde::to_vec` writes a payload as an array in declaration order), so a new field is only backward compatible at the tail; `additive_fields.rs` proves a pre-field `ReducerCall` frame still decodes with the key defaulting to `None` (an old client simply opts out). The roundtrip proptest generates the field.
- [x] 1.2 Define the durable dedup record (key, `(Identity, reducer)` scope, original `ReducerResult`, timestamp) and its bounded window config (count/age) (CS-030, CS-031; crates/fluxum-core/src/reducer/mod.rs) — new `reducer::idempotency`: the `__idempotency__` system table (`identity`, `reducer`, `key`, `created_us`) with composite PK `(identity, reducer, key)`, plus `IdempotencyOptions { max_records: 100_000, max_age: 1h }`. **The record stores no outcome payload**, because only committed calls are ever recorded (see the notes below) and a committed call's `ReducerResult` outcome is always `Ok(())` — storing a fixed value would be dead weight. The replay answers `Ok` with the *new* request's `id`.
- [x] 1.3 Add a pre-execute dedup check in the reducer dispatch path: on a hit, return the stored `ReducerResult` without invoking the reducer body (CS-030; crates/fluxum-core/src/reducer/engine.rs) — `ReducerEngine::call_idempotent(caller, name, args, key) -> Result<CallOutcome>`; a hit is `CallOutcome::Deduplicated` and the body never runs. The check happens **inside the reducer's transaction**, on the shard's single writer, so check-then-act is atomic: two concurrent calls with the same key cannot both miss. A hit aborts the transaction rather than committing an empty one, so it consumes no tx id (TXN-030). `key: None` is an ordinary call. The session routes `ReducerCall.idempotency_key` through it and publishes no diff for a dedup hit (the original commit already fanned out).
- [x] 1.4 Persist the dedup record as part of the reducer commit so it survives crash/restart (CS-031; crates/fluxum-core/src/txn/mod.rs, crates/fluxum-core/src/reducer/engine.rs) — the record is written into the reducer's own `Tx`, so it commits atomically with the effects it guards and rides the commit log. No change to `txn/mod.rs` was needed: riding the ordinary commit path *is* the durability. `the_dedup_record_is_written_in_the_reducer_commit` asserts one `TxDiff` carries both the effect and the record.
- [x] 1.5 Prune the dedup window by count/age from the schedule worker (CS-031; crates/fluxum-core/src/scheduler/mod.rs) — `prune_idempotency` runs from the schedule-worker loop on its own coarse cadence (`SchedulerOptions::idempotency_prune_interval`, default 30 s; rescanning on the 5 ms poll interval would be waste). It reads the committed window, decides via the pure `idempotency::prunable` (age bound, then oldest-surplus over the count bound), and deletes in one transaction. A shard without the table is a no-op.
- [x] 1.6 Scope keys strictly per `(Identity, reducer)` so distinct callers/reducers never collide (CS-031; crates/fluxum-core/src/reducer/mod.rs) — the scope *is* the composite primary key, so it cannot drift from the lookup. `keys_do_not_collide_across_callers` shows Alice and Bob both using key `"1"` each apply, while each caller's own replay is still caught.
- [x] 1.7 Ensure the SDK offline queue attaches a stable `idempotency_key` to every queued call for safe reconnect replay (CS-032; sdks/rust) — **partially: the key discipline ships, the socket/persistence cannot yet.** `sdks/rust` is a placeholder crate whose connection, cache and durable queue are DAG T6.2 / `phase6_sdk-optimistic-mutations-offline-queue`, both sequenced *after* the G5 wire freeze. Shipped now is `OfflineQueue`: it mints the key **once at enqueue** (never at send time — minting per attempt would defeat the mechanism entirely), keeps it stable across every retry, namespaces keys per client, and treats a deduplicated replay's result as an ordinary ack. Wiring it to a real socket + durable storage is carried as a follow-up on `phase6_sdk-optimistic-mutations-offline-queue`.

**Design notes / limits (deliberate)**
- **Only committed calls are deduplicated.** A reducer that returns `Err` or panics rolls its transaction back — and the dedup row, written in that same transaction, rolls back with it. This is the honest consequence of 1.4's atomicity, not a gap: a failed call applied nothing, so its retry re-executing is safe. The guarantee is "an applied call applies once", not "a failed call is remembered". Asserted by `a_failed_call_records_nothing_so_its_retry_re_executes`.
- **A key is a bounded safety net, not an indefinite promise.** A key pruned before its retry arrives is executed again (`a_pruned_key_is_executed_again`). CS-031 mandates the bound; this is what it costs.
- **A key on a shard without `__idempotency__` is an error, not a silent downgrade** — a client that asked for exactly-once is told, never quietly given at-least-once.
- Deployments accepting `idempotency_key` must include `IDEMPOTENCY_TABLE` in the assembled schema, like `__schedule__`.
