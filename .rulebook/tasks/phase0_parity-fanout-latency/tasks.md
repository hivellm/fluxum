## 1. Implementation
- [x] 1.1 Instrument before fixing — **`fluxum_fanout_stage_us{stage}` histogram landed (OBS-023)**: recv_lag (commit broadcast→fan-out wake), eval (manager lock + on_commit), enqueue (encode + all try_sends), queue_wait (per-frame enqueue→writer dequeue, `OutFrame` now carries its enqueue instant), flush (per-frame write_all), server_total. **Measured 2026-07-22, release, 50 subscribers @ 10 msg/s (client-observed p50 498 µs / p99 771 µs)**: recv_lag 12.7 µs mean, eval 13.1 µs, enqueue 36.5 µs, **queue_wait 107.5 µs** (the largest server stage: 50 writer-task wakes from one worker + steal rounds), **flush 46.1 µs/frame** (Windows loopback WSASend under 50-way concurrency), server_total 73.8 µs mean / p99 ≤ 250 µs. Fan-out slope by subscriber count: 1 sub p50 310 / p99 484 µs; 10 subs 341/542; 50 subs 498/771 — the 1→50 slope is +188 µs p50 / +287 µs p99 (delivery serialization), the **base** (~310 µs) is commit leg (~write RTT p50 161 µs) + single delivery (~110 µs) + SDK reader OS thread wake (~50-150 µs tail, structural at 10 msg/s — the reader is always parked when the frame lands, so spin-before-park cannot help sparse traffic). Driver-side share isolated by core separation (see 1.4): sharing cores inflated p50 by ~104 µs / p99 by ~85 µs
- [x] 1.2 If server-bound: make bucket fan-out concurrent/batched — **tried and honestly reverted**: chunked parallel enqueue (4 spawned delivery tasks per delta, per-subscriber ordering held by chunk-partitioning + per-delta join) measured NO better than the inline loop (p99 734 µs vs 686 µs, both ±85) and risks a convoy behind a starved chunk task; the enqueue loop is 36 µs — the cost is the writers' scheduler dispatch + 50 individual socket writes (46 µs each, a syscall floor), not the enqueue. Kept: the once-encoded shared `Arc` bytes (already there), the `deliver_frame` extraction, and the OBS-023 instrumentation to keep this measurable. A direct-socket write path (bypassing the writer task for plaintext) was then scoped and **discarded by arithmetic**: inlining 50 sequential `try_write`s in the fan-out task (50 × ~10-15 µs) serializes what the parallel writers spread across all 32 workers (~150 µs wall) — it cannot beat the floor it replaces. The measured floor on this box is `subscribers × 46 µs Windows-loopback write / workers` + dispatch: **environmental, not architectural** (Linux write syscalls are ~5-10 µs; see the WSL2 floor measurement in 1.5)
- [x] 1.3 If server-bound (contention half, F-006): take fan-out off the commit critical path — **DONE via `CommitHook` on `TxPipeline` (TXN-021 steps 9/10, which the spec always prescribed as concurrent)**: the single writer publishes every diff to the shard/namespace broadcast at commit visibility — before the log handoff, the written watermark, and the caller's oneshot→transport-task response hop; `ShardContext::with_views`/`Namespace::with_quotas` install the hook (Weak-bound, first-install-wins), every caller-side `publish_commit` was removed (tcp/http/admin/sweepers; replication's direct-store apply keeps its manual publish). Root cause of the mixed tail: publish rode BEHIND the ack's `wait_written`, so the log writer's periodic fsync stalled delivery ~50 ms. **Measured (CLI defaults, 8w+8r+50 subs): mixed/e2e p99 45.9 ms → 0.728 ms (63×), max 50.6 → 1.06 ms, p50 0.496 → 0.391 ms; mixed/write p99 1.67 → 1.60 ms, +9% throughput; standalone e2e p99 0.783 → 0.726 ms**. Ordering improved (worker publishes in tx_id order — transports could race), ack durability unchanged (TXN-004 `wait_written` stays on the response path); a subscriber may now see a commit the documented ~50 ms crash window erases, healed by SPEC-021 resync — same trade respond-after-merge already made. Priority admission lanes (the "and/or") not needed: the queue-wait half was never the tail. Full workspace suite green (129 targets)
- [x] 1.4 If driver-bound: fix the harness measurement — **partially driver-bound, fixed by core pinning**: the driver's 50 subscriber reader threads shared cores with the server's runtime and inflated fluxum e2e p50 by ~104 µs / p99 by ~85 µs. `fluxum-bench` grew `--pin server=0xMASK,driver=0xMASK` (Windows: PowerShell `ProcessorAffinity`; Unix: `taskset`): the driver self-pins, and EVERY spawned server (fluxum-server and the incumbent's baseline-server alike — fairness) is re-pinned to the server mask; containerized servers live on the Docker VM's cores. The active spec is recorded in the report's stack config strings (TST-091: methodology is part of the published configuration). **Full-report measurement (2026-07-22): pinning 16/16 HURTS the matrix on this box** — the server confined to 16 logical cores degrades every heavy phase (fluxum e2e p99 758 µs unpinned → 1 061 µs pinned; mixed/e2e 4.5 ms → 22.2 ms; PG's app server degrades too, e2e p99 6.48 → 7.30 ms) while helping only the isolated warm-server e2e (−85 µs). The canonical report therefore runs UNPINNED; `--pin` stays as a documented methodology knob for isolated-latency investigation. Server-emit timestamping was evaluated and REJECTED: TST-092(b) as measured (send→receipt) includes each side's own commit leg symmetrically; switching to commit→receipt would drop PG's ~5.5 ms INSERT from its number and shrink the honest gap — the current methodology is the conservative one
- [x] 1.5 Verification (exit test): **3 consecutive full reports run 2026-07-23** (PG 17 + SpacetimeDB v2.6.1 in Docker, fluxum native, unpinned, machine quiet). **mixed/e2e: decisively met, 3/3** — fluxum p99 0.664–0.683 ms vs PG 13.0–15.4 ms = **19.3× / 20.3× / 22.6×** (was 1.68×; F-006 closed). **e2e_p99 vs PG: 5.25× / 9.56× / 10.8×** — fluxum's own p99 is reproducible (0.676–0.745 ms, ±5%); the ratio's spread is PG's run-to-run p99 swing (3.91 → 7.30 ms, within-run σ up to 1.5 ms).
