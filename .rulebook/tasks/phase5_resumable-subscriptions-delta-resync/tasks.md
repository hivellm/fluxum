## 1. Implementation

- [x] 1.1 Add a monotonic per-shard `tx_offset` field to `InitialData` and `TxUpdate` (mirroring `tx_id`) and confirm it stays additive/positional (CS-020, CS-023; crates/fluxum-protocol/src/messages.rs) — both carry `#[serde(default)] pub tx_offset: u64`, appended last-but-one before `tables` so pre-field frames still decode. It mirrors `tx_id` today and is documented as *the* cursor clients retain, so the offset space can diverge from tx ids later without breaking resumption.
- [x] 1.2 Add the additive `Resume { id, query_id, from_offset }` client message struct and register it in the `ClientMessage` tagged enum (CS-021, CS-023; crates/fluxum-protocol/src/messages.rs) — registered as the `"Resume"` tag (RPC-026) and re-exported; the roundtrip proptest generates the variant.
- [x] 1.3 Track the highest committed `tx_offset` per subscription and retain a bounded delta window (by count/age) in the subscription manager (CS-020, CS-021; crates/fluxum-core/src/subscription/mod.rs) — `last_offset` advances on every commit (whether or not a query matched); each query bucket keeps a `DeltaWindow` of the last `SubscriptionLimits::resume_window_deltas` updates (default 256, **by count**; an age bound is not implemented — see the note below). Both are interior-mutable, since `on_commit` takes `&self`. The window is freed with its bucket.
- [x] 1.4 Implement resume replay: given `from_offset` inside the retained window, emit only committed deltas after it for the compiled query, then resume live `TxUpdate`s (CS-021; crates/fluxum-core/src/subscription/mod.rs) — `SubscriptionManager::resume` returns `Resumed::Deltas` (ascending by offset; empty when already caught up); the session ships them as `TxUpdate`s stamped with the offset each committed at, and the ordinary fan-out carries on live.
- [x] 1.5 Implement the compacted-window fallback: when `from_offset` predates the retained window, send a full `InitialData` plus a cache-reset signal (CS-022; …) — a resume is serviceable iff `from_offset >= trimmed_before` (the highest evicted offset); otherwise `Resumed::Reset` rebuilds the bucket's snapshot for its viewer with the additive `InitialData::cache_reset` flag set. It never silently under-replays.
- [x] 1.6 Route the `Resume` message through the transports and session layer, preserving per-subscription offsets across GET reconnects (CS-021; crates/fluxum-server/src/{session.rs,http.rs,tcp.rs}) — routed in the transport-independent `session.rs` (so both transports carry it); an unknown `query_id` is the new 6002 `SUB_UNKNOWN_QUERY_ID` telling the client to subscribe afresh. **This item required a real fix, not just routing:** the GET teardown previously evicted the session and called `disconnect()`, destroying the bucket and its window — `Resume { query_id }` could never have resolved after a blip, so the feature would have been inert. A dropped GET stream now only *detaches* (the receiver is handed back, session/subscriptions/window survive); idle expiry, server shutdown and a fan-out drop still retire the session and run `on_disconnect`, and a new session sweeper owns RPC-060 expiry for sessions no stream is timing so an absent client cannot pin subscriptions forever. **Scope:** resumption therefore serves the HTTP Streamable GET-reconnect case. A dropped TCP socket still ends its connection and subscriptions as before, so a reconnecting TCP client gets 6002 and re-subscribes.
- [x] 1.7 Have the Rust SDK retain the highest applied `tx_offset` per subscription and send `Resume` on reconnect, handling the cache-reset fallback (CS-020, CS-022; sdks/rust) — **partially: the bookkeeping ships, the socket send cannot yet.** `sdks/rust` is a placeholder crate; its connection, cache and codegen are DAG **T6.2** (`phase6_rust-sdk`), explicitly sequenced *after* the G5 wire freeze, so there is no client to send from. Shipped now is `ResumeTracker` — the part the wire freeze actually constrains: retains the highest applied offset per `query_id` (monotonic; a replayed update cannot rewind it), folds in `InitialData`/`TxUpdate`, surfaces the CS-022 `cache_reset` as a must-clear signal that re-anchors the cursor, and decides `Resume` vs `Subscribe` on reconnect (`resume_message` yields the ready `ClientMessage`). Wiring it to a real socket is carried as a follow-up on `phase6_rust-sdk`.
- [x] 1.8 Confirm all wire additions are additive and sequenced to land before the G5 wire freeze (CS-023; crates/fluxum-protocol/src/messages.rs) — every added field is `#[serde(default)]` and appended, never reordered (the envelope is positional); `Resume` is a new tag, and 6002 is a new catalog code with `docs/errors.md` regenerated from the catalog. Landed now, ahead of the freeze. The roundtrip proptest exercises `tx_offset`, `cache_reset`, `shard_id` and `Resume`.

**Not implemented / follow-ups**
- The delta window is bounded **by count only**; 1.3's parenthetical "by count/age" age bound is not implemented (nothing in CS-020/021 requires it, and the count bound alone caps memory).
- A window stays complete only while its bucket has a subscriber — i.e. exactly the retained-session case 1.6 enables. It is not a general offset log.
- SDK socket wiring (send `Resume` on a real reconnect, apply the reset to a row cache) → `phase6_rust-sdk` (T6.2).
