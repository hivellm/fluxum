## 1. Implementation
- [x] 1.1 Lexer + parser: add <, >, <=, >= as top-level AND conditions; keep OR/!=/LIKE/NULL rejected with the existing 400 diagnostic (QP-030; sql/lexer.rs, sql/parse.rs) — `Token::{Lt,Le,Gt,Ge}` (two-char lookahead), `CondAst::Cmp(col, CmpOp, lit)`; `!=`/`<>`/OR/LIKE/NULL/IS stay named 400s (injection corpus updated: the four operators moved from HOSTILE to the legal subset, negations remain hostile)
- [x] 1.2 Fold a same-column range pair (col >= a AND col <= b) into one closed/half-open interval equivalent to BETWEEN for pushdown (QP-030) — `plan_access` intersects every BETWEEN + comparison on the range-position column via `tighter_lower`/`tighter_upper` (greater/smaller value wins; Excluded beats Included at equal values); strictness preserved (`>` stays `Bound::Excluded`)
- [x] 1.3 Type-check comparison operators via existing schema-typed coercion (int->float widening); reject on Bool/Option/List at compile time (QP-032) — `Cmp` compile arm reuses `coerce` and the BETWEEN type discipline; `flag > TRUE` pinned as a QP-032 400
- [x] 1.4 Parse the AFTER (value, pk) cursor clause; allow only with ORDER BY on an indexed column (QP-040) — `AFTER (lit, lit)` after LIMIT; parser rejects AFTER without ORDER BY; compile rejects it when the order is not index-served (`ordered_by_index`); OFFSET is rejected with a pointer to keyset (QP-040: linear + unstable, deliberately absent)
- [x] 1.5 Append primary key as the final ORDER BY term (implicitly if omitted) for a total order and unambiguous cursor (QP-041) — an explicit `, pk [ASC|DESC]` tiebreak parses and must name the single-column PK with the matching direction (400 otherwise); implicit otherwise. The total intra-tie order is the PK's index byte order — the order the B-tree actually yields within one key, so walk and cursor comparison share one total order (found via a real overlap bug: numeric-PK comparison diverged from the index's encoded-byte order; the fix compares encoded PK bytes)
- [x] 1.6 Translate AFTER cursor into an index lower bound (value > c OR (value = c AND pk > k)) through the planner access path; no OFFSET (QP-040) — compile coerces both cursor values against the order/PK column types, tightens the scan's lower (ASC) / upper (DESC) bound to Included(c), and the evaluator skips the `value = c AND pk ≤ k` boundary residue in-scan by encoded-PK comparison (`pk ≥ k` on the DESC reverse walk); cursor rides `CompiledPlan.cursor` (the slot the planner task reserved) and changes the QueryHash (each page is its own normalized query)
- [x] 1.7 Pagination applies to snapshot (InitialData/one-off) only; subsequent pages via new SubscribeSingle/OneOffQuery with advanced cursor (QP-042) — the cursor executes exclusively in the `initial_data` index-scan path; TxUpdate delta evaluation is untouched
- [x] 1.8 Reflect extended operator set in SDK codegen + /schema query documentation (QP-031; SPEC-011) — before T6.1 freeze — `/schema` now carries a `query` section: `operators: ["=", "IN", "BETWEEN", "<", ">", "<=", ">="]` + the keyset pagination shape (the surface SDK codegen consumes; the SDKs themselves are phase-6 tasks); explain reports the cursor
- [x] 1.9 Verification: paging a large table returns each row exactly once, no gaps/overlaps under a stable snapshot; page N+1 rows-scanned approx page size independent of N; range operators push down and return correct sets — 500 rows with 10-way ties paged at 37/page in BOTH directions: exactly-once, 14 pages, order monotone; counters: deep page (page 8) scans ≤ first page + 1; operator pushdown pinned structurally (bounds) and by executed row sets
