## 1. Implementation
- [x] 1.1 SDK: support multiple in-flight reducer calls per connection — **`Connection::call_reducer_async` → `PendingReducer::wait`** (the wire already correlates by request id, RPC-002; the sync `call_reducer` is now async+wait). Attribution exact (each handle resolves its OWN ack/error — proven by a doomed call planted mid-window failing alone); per-connection ordering preserved (sends serialize on the write half, the server executes in arrival order — proven by 8 pipelined adds delivering in submission order); disconnect fails every in-flight handle; abandoned handles clean their pending entry on Drop. Contract documented in rustdoc + normative **SDK-032** added to SPEC-011: no client-side cap — backpressure is the TCP send buffer + TXN-011 shard-busy (which resolves exactly the refused calls); over Streamable HTTP the POST itself round-trips, so pipelining is a TCP-transport property
- [x] 1.2 Bench: `--pipeline N` on the write workload (default 1 = acked-serial, unchanged); pipelined runs are a **separate class** `write/pipelined(N)` — never the same label. `BenchClient` grew `start_task`/`finish_task` (default: unsupported — a request/response side's concurrency lever is `--clients`, already measured). The report gains a fluxum-only `write/pipelined(32)` row that **feeds no ratio**, with a markdown footnote (latency includes the deliberate window queueing; throughput is the meaningful column; no baseline counterpart by protocol design)
- [x] 1.3 Sweep on the parity box (release, 2026-07-23, machine quiet): **clients=8 × N**: 1 → 39 967 ops/s (p99 1.55 ms) | 8 → 63 484 | 32 → 64 521 | 128 → 62 003 — **the curve flattens at N=8, ~64k ops/s**; larger windows only inflate queueing latency (p50 0.71 → 4.3 → 16 ms). **N=32 × clients**: 1 → 19 059 | 4 → 52 934 | 8 → 64 521 | 16 → 63 279 | 24 → 63 469 — aggregate invariant beyond 8 connections. Report row pinned at window 32 (`REPORT_PIPELINE_WINDOW`, past the knee)
- [x] 1.4 Verification (exit test): **100k NOT demonstrated — the ceiling is measured and attributed**: the wall is the **shard's single-writer commit loop at ~15.6 µs/commit sustained** (64k/s aggregate, invariant to connection count 8→24 and window 8→128 — eliminating network, driver, and per-connection read loops; reducer queue never full — no shard-busy at 512 in flight vs the TXN-011 default 1000). Cross-check: `txn_commit` engine microbench (no sockets) runs the same cycle at **22.4 µs serial round-trip** (p50 22.7, includes the caller's oneshot wake that pipelining overlaps); `fluxum_reducer_duration_us` under saturation reads 175 µs mean = queue residency (window × service), matching client p50 8.6 ms at 512 in flight. Pre-task the cap was the PROTOCOL (acked-serial 1/RTT: 40k @ 8 conns, 4.5k single conn); post-task it is the ENGINE: T6.6 starts from "single-writer loop, 15.6 µs/commit — execute+merge+timestamp+ephemeral-scan+commit-hook+log-acceptance+oneshot-send; needs 1.56× to NFR-01's 10 µs", with batch-drain of the worker loop (amortize channel recv + timestamp across queued jobs) as the named next lever. Recorded in SPEC-013 TST-060 (which now mandates the pipelined methodology)
