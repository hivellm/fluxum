## 1. Implementation

- [x] 1.1 Define the fixture file format (rows and reducer calls) and its loader/parser (DEV-040; crates/fluxum-cli) — `seed.rs`: a JSON document of ORDERED reducer calls `{ "calls": [ { "reducer", "args", "repeat"? } ] }`. Rows are expressed as calls to the module's own reducers whose args carry the row data — the SPEC-024 §6 rule (all mutations go through reducers) applies to seeding too, so there is deliberately no direct-row section. Parser rejects empty call lists, blank reducer names and `repeat: 0` with positional messages; unit-tested.
- [x] 1.2 Implement `fluxum seed <file>` targeting a running or fresh instance (DEV-040; crates/fluxum-cli) — `fluxum seed <file> --server <host:port>`: one `POST /reducer/<name>` per call (repeat expanded) via the new shared `post_path` (the hand-rolled one-request HTTP shape refactored into `http_request`, now serving GET and POST). Non-200 answers surface the admin envelope's own `error` message. Stops at the first failure — fixtures are ordered, so skip-and-continue would seed a silent half-state.
- [x] 1.3 Apply seeded fixtures through reducers on the server side (DEV-040; crates/fluxum-server) — the admin surface's existing `POST /reducer/<name>` (RPC-051) IS the application path: full production admission (arg validation, rate limits, client-callable gating, the reducer body, the commit) runs for every fixture call; no seed-only bypass was added, by design.
- [x] 1.4 Expose the migration diff-planning path read-only from the migration module (DEV-041; crates/fluxum-core/src/migration/diff.rs, runner.rs) — `fluxum_core::migration::{plan, plan_with, MigrationPlan, PlanVerdict}`: snapshot reads only (does not even take the commit log), reusing `diff_catalogs_with` and the runner's validation; `run`'s `read_meta` now shares the same helper. Verdicts: FirstBoot / UpToDate / AutoApplies / NeedsVersionBump (MIG-023) / RequiresMigration (MIG-022) / RunsMigrations (pending steps run first — the plan is honest that a read-only preview cannot execute them, so the diff is against the CURRENT stored catalog). MIG-003 downgrade parity: the plan errors the way the boot would.
- [x] 1.5 Implement `fluxum migrate --plan` printing the computed schema diff without mutating state (DEV-041; crates/fluxum-cli, crates/fluxum-core/src/migration/) — the schema lives in the module BINARY, so the CLI builds the module (`cargo build`, the `fluxum dev` machinery) and runs its executable with `FLUXUM_MIGRATE_PLAN=1` — a seam inside `fluxum_server::boot::serve`, so every embedder (reference binary, `fluxum init` scaffolds, custom mains) gets plan mode with zero code changes; the reference binary also takes `--migrate-plan` directly. Child exit code propagated: 0 = boots, 3 = would refuse, 1 = plan failed.
- [x] 1.6 Classify and print the auto-apply decision (safe/additive vs. requires-migration) per diff entry (DEV-041) — `MigrationPlan::render()`: one line per entry with the MIG-020 `required_action()` and an `[auto]` / `[BLOCKS]` tag, pending steps listed in order, and a one-line verdict.
- [x] 1.7 Guarantee `migrate --plan` is side-effect-free (DEV-041) — by construction (no `Tx::begin`, no `CommitLog`), and pinned by tests: the core suite plans twice over a never-booted store and asserts `__schema_meta__` version/catalog are untouched after planning a pending v2; the server suite asserts a plan against a missing data dir reports first boot and CREATES NOTHING (no log dir, no checkpoint dir), for both `boot::migration_plan` and the spawned `fluxum-server --migrate-plan` binary.
